# ТЗ: React Task Manager

## Цель

Цель — **не красивый интерфейс, а понимание базовой логики руками**: `useState`, `useEffect`, `props`, события, списки, фильтрация, формы, `localStorage`.

По итогу нужно не просто получить работающее приложение, а уметь объяснить, как и почему оно работает (см. раздел «Что нужно уметь объяснить»).

---

## Что делаем

Небольшое приложение для управления задачами (тудушка). Без бэкенда, всё хранится в браузере.

---

## Функциональные требования

Приложение должно уметь:

1. Добавлять задачи через input и кнопку.
2. Не добавлять пустые задачи (пробелы тоже не считаются).
3. Показывать список задач.
4. Отмечать задачу выполненной через checkbox.
5. Удалять отдельную задачу.
6. Фильтровать задачи: **все / активные / выполненные**.
7. Показывать статистику: всего / выполнено / осталось.
8. Удалять все выполненные задачи одной кнопкой.
9. Сохранять задачи в `localStorage`.
10. После перезагрузки страницы задачи должны восстановиться.

```bash

cd task-manager
npm install

```

(Vite, не Create React App — CRA устарел.)

## Обязательные React-темы

В задаче должны быть реально использованы:

- `useState` — для задач, значения input и выбранного фильтра
- `useEffect` — для сохранения задач в `localStorage`
- `props` — для передачи данных и функций в дочерние компоненты
- `map` — для вывода списка
- `filter` — для фильтрации и удаления
- обработчики событий: `onChange`, `onClick`, `onSubmit`

---

## Структура компонентов

Минимальное разделение:

- `App` — хранит основной state (массив задач + текущий фильтр), главный компонент состояния.
- `TaskForm` — добавление новой задачи.
- `TaskList` — получает список и выводит его.
- `TaskItem` — одна задача (текст, checkbox, кнопка удаления).
- `TaskFilters` — переключение фильтров.
- `TaskStats` — статистика.

Состояние живёт в `App`, вниз передаётся через `props` (данные + функции-обработчики).

### Модель одной задачи

```js
{
  id: Date.now(),
  text: 'Пример задачи',
  completed: false
}
```

## Git и ветки

Постоянные ветки:
- `master` — стабильная, «релизная». Напрямую туда ничего не пушим.
- `dev` — основная ветка разработки, в неё вливаются все задачи.

Под каждую задачу — отдельная ветка от `dev`:

```
feature/setup
feature/add-tasks
...
```

Закончил задачу → открываешь PR из `feature/...` в `dev`. После ревью — мёрж. В `master` `dev` вливается отдельно, когда набралась рабочая версия.

Правила:
- один PR = одна задача, маленький и читаемый;
- осмысленные коммиты по ходу работы, а не один коммит «done»;
- напрямую в `master` и `dev` не пушим — только через PR.

---

## План работ (каждая задача = ветка + PR)

Идти по порядку, **каждую задачу сначала руками**, не сразу всё. Внутри задачи делай промежуточные коммиты по подпунктам.

1. `feature/base` — Vite-проект и базовая структура компонентов; форма, добавление задачи и вывод списка (`useState` + `map`). Пустые задачи не добавляются.
2. `feature/task-actions` — удаление задачи через `filter` и переключение `completed` через checkbox.
3. `feature/filters-stats` — фильтры (все / активные / выполненные, `useState('all')`), статистика и кнопка «удалить выполненные».
4. `feature/persist` — сохранение в `localStorage` через `useEffect` и восстановление при загрузке. Сюда же финальная чистка структуры после ревью.

Каждую задачу закрываешь отдельным PR в `dev` — так набивается ритм «ветка → PR → ревью → мёрж».

## Как использовать AI

**Можно:**
- объяснить непонятную тему;
- подсказать идею;
- помочь найти ошибку;
- спросить, почему `useEffect` сработал несколько раз;
- проверить уже написанный код;
- предложить улучшение структуры, **когда база уже написана руками**.

**Нельзя:**
- «сделай мне весь проект» / «напиши всё приложение целиком»;
- «сгенерируй все компоненты сразу»;
- копировать код, не понимая, как он работает.

Правило простое: если не можешь объяснить строчку — её писать рано.

## Definition of Done (когда задача принята)

- [ ] Все 10 функциональных пунктов работают.
- [ ] Пустую/из пробелов задачу добавить нельзя.
- [ ] Фильтры переключаются и корректно показывают подмножество.
- [ ] Статистика совпадает с реальным состоянием списка.
- [ ] После F5 задачи на месте.
- [ ] Состояние не мутируется напрямую (никаких `tasks.push(...)`).
- [ ] Приложение разбито на компоненты из раздела «Структура».
- [ ] Есть README с командами запуска проекта.
- [ ] Каждая задача прошла через свой PR в `dev`, код проходит ревью-разговор (ниже).

---

## Опционально

Только если база готова, строго «по желанию» и не в ущерб основному:

- редактирование текста задачи по двойному клику;
- адекватный вид при пустом списке;
- кнопка/поведение сброса фильтра;
- сортировка или дата создания.

---

## Что нужно уметь объяснить на ревью

Финальная приёмка — не «работает/не работает», а разговор. Будь готов словами объяснить:

- где хранится список задач и почему именно там;
- как добавляется новая задача;
- **почему нельзя мутировать массив напрямую**;
- как работает `map`;
- как работает `filter`;
- зачем нужен `useEffect`;
- как данные сохраняются и восстанавливаются из `localStorage`;
- как `props` передаются в компоненты;
- почему `App` — главный компонент для состояния.
